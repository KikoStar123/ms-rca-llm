## 1. 研究背景与问题定义
### 1.1 背景与痛点
微服务架构具有调用链长、组件依赖复杂、版本频繁变更的特点。故障发生时，日志、指标、调用链追踪构成多源观测数据，信息量大但噪声高。传统基于规则或经验的排障方式存在以下问题：
- 证据分散且难对齐，难形成稳定结论；
- 依赖链路动态变化，规则维护成本高；
- 排障结果缺乏可解释性与一致性，难以复用。

### 1.2 问题定义
目标是设计一种“大小模型协同”的根因定位与故障类型分类方法，使系统能够：
- 在多源观测数据中提取关键证据；
- 输出可解释的根因组件与故障类型；
- 形成可用于大模型训练的数据样本；
- 具备工程化落地的端到端流程。

### 1.3 研究目标与约束
研究目标对应三个核心阶段：
1) 数据准备与 SFT：构建结构化样本，完成监督微调准备。  
2) GRPO 优化：在 SFT 基础上提升输出稳定性与可靠性。  
3) 系统工程实现：完成数据接入、流程编排、结果输出与展示。

## 2. 技术路线与总体框架
### 2.1 总体流程与数据流
整体流程分为六个核心环节，按“数据 → 证据 → 候选 → 推理 → 结构化输出”串联：
1) 数据接入与预处理  
2) 时间窗口对齐与证据聚合  
3) 小模型生成 Top‑K 根因候选  
4) 大模型推理（根因解释与类型分类）  
5) 统一输出 Schema 与校验  
6) 结果展示与系统编排

### 2.2 大小模型协同原则
大小模型协同遵循“先缩小搜索空间，再做解释推理”的原则：
- 小模型：快速处理大规模数据，生成 Top‑K 候选根因，提供搜索约束；
- 大模型：基于证据与候选集合进行推理，输出根因、类型与解释；
- 统一模板与 Schema：保证可验证性、可复现性和工程输出稳定。

**为什么这样做**：
- **搜索空间过大**：微服务组件众多，直接让大模型在全量组件中找根因，容易产生“猜测式结论”。先用小模型给候选集合，相当于把搜索空间收缩到可控范围。
- **资源与效率**：小模型做召回，成本低且速度快；大模型专注在小集合上解释和分类，推理成本显著降低。
- **稳定性与可解释性**：候选集合让大模型“有据可依”，输出更稳定，也更容易对应到证据摘要。
- **工程可评估**：Top‑K 有明确的评价指标（Hit@K），能量化小模型召回能力，为整体系统提供可量化基线。

### 2.3 关键设计取舍
- 证据聚合采用统计摘要而非原始日志，以降低训练噪声与输入长度；
- 使用 Top‑K 候选约束 LLM 推理，提高准确率与一致性；
- 通过 Schema 校验保障“可交付”的结构化输出。

**为什么这样取舍**：
- 原始日志与链路跨度大、噪声高，直接输入会导致上下文过长、训练不稳定；统计摘要能保留“异常趋势”而不引入过多噪声。
- Top‑K 机制在“召回优先”的前提下能显著提升 LLM 推理稳定性，避免输出超出合理范围。
- Schema 校验让输出成为工程可用结果，而非“自由文本”，便于对接 API、前端与后续评估。

## 4. 数据准备与样本构建
### 4.1 时间窗口对齐
核心思想：围绕 `log_time` 构建固定时间窗口，在该范围内汇聚多源证据，避免无关噪声。
- 窗口参数：`window_before`、`window_after`（以分钟为单位或脚本内固定换算）；
- 统一窗口使跨源数据能对齐到同一个事件语义。

**为什么这样做**：
- 故障往往呈现“前后波动”，只看单点时间容易错过关键异常。
- 固定窗口能把日志、指标、调用链统一切片，形成可复现的证据范围。
- 同一策略可复用于所有事件，便于批量生成样本与统计。

### 4.2 证据聚合策略与细节
证据按数据源分三类汇总为统计摘要：
1) 平台指标：`count / avg / min / max`  
2) 调用链指标：`count / avg_elapsed / success_rate`  
3) 业务指标：`count / avg_time / avg_success_rate / total_num`

聚合逻辑采用“单日文件级一次读取、事件级多路累积”的策略：
- 每个 CSV 文件只扫描一次；
- 对同一天多个故障事件同时更新聚合结果；
- 避免重复 I/O，显著降低时间成本。

**为什么这样做**：
- 证据摘要能显著压缩输入长度，避免 LLM 在大量原始记录中迷失重点。
- count/avg/min/max 等指标能保留“异常趋势”和“波动范围”，是根因判断的最小充分信息。
- 单日一次读取是出于性能考虑，避免对多事件重复扫描数百万行数据导致极慢。

### 4.3 SFT 样本生成流程
脚本：`pipeline/build_sft_samples.py`  
主要步骤：
1) 读取 `故障整理（预赛）.csv` 并按日期筛选事件；
2) 为每个事件构造时间窗口；
3) 读取平台/业务/链路指标并完成聚合；
4) 输出 JSONL 样本 `output/sft_samples_YYYY_MM_DD.jsonl`。

样本结构字段：
- `fault_event`：事件元信息；
- `evidence`：聚合后的证据摘要；
- `output`：根因组件与故障类型（作为训练标签）。

### 4.4 批量构建与统计
批量脚本：`pipeline/batch_build_sft.py`
- 自动解压每日数据；
- 按日期生成样本；
- 输出每天的统计与汇总信息。

统计脚本：`pipeline/sft_stats.py`
- 统计样本总量、故障类型分布、根因组件分布；
- 用于判断训练数据规模与分布均衡性。

## 5. 小模型基线与根因 Top‑K
### 5.1 故障类型分类基线
脚本：`pipeline/small_model_baseline.py`
- 构建频次映射：`(object, kpi) -> fault_type`；
- 采用“最新日期”为测试集，其余日期为训练；
- 输出基线准确率，作为简单对照。

### 5.2 根因候选 Top‑K（基础版）
脚本：`pipeline/small_model_rootcause_topk.py`
核心逻辑：
- 对窗口内 `cmdb_id` 计数；
- 按出现频次排序，输出 Top‑K；
- 使用 Hit@K 评估候选覆盖率。

**为什么用 Top‑K**：
- 根因空间很大，Top‑K 先保证“召回”，再由 LLM 精判。
- Hit@K 可以直接衡量候选覆盖能力，是工程上最直观的基线指标。

### 5.3 根因候选 Top‑K（加权版）
脚本：`pipeline/small_model_rootcause_weighted_topk.py`  
改进要点：
- 平台指标引入异常强度：使用数值范围与均值比例作为异常分数；
- 调用链引入时延与失败惩罚：`elapsedTime` 越大、失败越多权重越高；
- `rate_mode` 计算失败率/慢调用率；
- `anomaly_once` 仅在事件级追加异常分数，避免重复计数。

**为什么引入加权**：
- 仅靠频次容易把“高频但非根因”的组件排到前面。
- 异常强度与失败惩罚更贴近“异常即根因”的经验假设。
- `rate_mode` 把绝对量转成比例，避免样本量差异造成偏置。

批量评估与对比：
- `pipeline/batch_rootcause_weighted_topk.py`：多日评估；
- `pipeline/compare_topk_summaries.py`：不同方法对比；
- `pipeline/grid_search_topk.py`：参数搜索获取更优权重。

## 6. LLM 输入构建与训练准备
### 6.1 LLM 输入结构设计
参考文档：`research/大模型输入模板.md`  
结构分为三部分：
- `instruction`：统一任务描述，控制模型输出风格；
- `input`：故障事件 + 证据摘要 + Top‑K 候选；
- `output`：根因、故障类型与解释，作为监督目标。

**为什么采用这种结构**：
- 通过统一 instruction 控制模型输出格式，避免自由发挥。
- 将证据与候选放在 input，保证推理有据可依且范围受控。
- 将目标输出显式结构化，方便训练与后续 Schema 校验。

### 6.4 大模型选择与使用说明
当前工程侧重点是“数据与流程可复现”，并未在本地完成真实的大模型训练或部署，因此这里描述的是**推荐选择与可落地方案**，而非已固定的单一模型。

推荐模型类型与理由：
- 7B~14B 量级指令微调模型：更易在单机或有限显存环境中开展 SFT/LoRA 实验；
- 支持中文与指令跟随能力：保证对中文故障描述与结构化输出的稳定性；
- 社区生态完整：便于使用成熟的训练与推理工具链。

可选模型示例（均为可替换方案）：
- Qwen2.5‑7B / 14B（中文能力强，指令跟随稳定）
- Llama3.1‑8B（英文较强，中文可用，生态丰富）
- ChatGLM3‑6B（中文表现稳定，推理成本低）

**说明**：实际选型应结合显存、训练预算与推理速度目标决定。本项目当前产出聚焦在“输入/输出结构、训练样本与工程流程”，大模型可以在后续训练阶段灵活替换。

**为什么不直接固定模型**：
- 训练资源与显存条件不确定，强行指定会影响可落地性。
- 本阶段目标是“把可训练样本与流程做完整”，模型可在后续按资源替换。
- 这样可以保证报告真实性，不假设未完成的训练结果。

### 6.5 推理阶段提示词与输出约束
大模型输出稳定性主要通过三类约束保证：
1) **统一输入模板**：将 `fault_event`、`evidence`、`top_candidates` 明确结构化；
2) **统一输出 Schema**：要求固定字段与数据类型；
3) **样本内对齐**：训练样本中强制对齐“证据引用与结论一致”。

**为什么需要约束**：
- LLM 对格式高度敏感，不加约束会产生难以解析的自由文本。
- 结构化约束能提升一致性，便于后续系统读取与展示。

建议推理提示词（简化版逻辑）：
- 任务：从证据与候选中确定根因组件与故障类型；
- 输出：严格 JSON，字段名与 Schema 一致；
- 解释：必须引用 `evidence` 中的关键统计或异常指标。

### 6.6 LLM 与小模型协作细节
协作方式为“先筛选、再解释”：
- 小模型输出 Top‑K 候选作为 LLM 可选集合；
- LLM 在候选集合内决策，避免“凭空猜测”；
- 若 LLM 推理超出候选，可引入 `other_candidate` 保留兜底路径。

**为什么要限制候选范围**：
- 防止模型输出与证据无关的组件，提升结论可信度。
- 通过候选约束，使 LLM 专注“解释与分类”，而非“枚举与猜测”。

### 6.7 LLM 稳定性与 GRPO 目标
GRPO 训练设计目标：
- **正确性**：根因与类型必须与标签一致；
- **一致性**：多次推理结果稳定；
- **可解释性**：解释必须引用证据字段；
- **格式合规性**：严格符合 JSON Schema。

在实际训练时，可将这些目标转化为奖励项进行加权，逐步提升模型输出可靠性。

**为什么需要 GRPO**：
- SFT 能让模型学会“输出样式”，但对稳定性与一致性提升有限。
- GRPO 通过奖励约束，强化模型在证据引用与格式方面的可靠性。
