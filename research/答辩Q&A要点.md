# 答辩Q&A要点（常见问题与回答要点）

更新时间：2026-01-23

## 1. 选题与意义
**Q1：为什么选择微服务 RCA？**
- 微服务依赖复杂、故障传播链长，传统规则难以快速定位。
- 多源观测数据丰富但噪声高，需要可学习、可解释方法。

**Q2：课题的核心目标是什么？**
- 设计并实现“大小模型协同”的根因定位与故障类型分类系统。
- 建立可用于 SFT/GRPO 的数据流程，并验证端到端可用性。

## 2. 方法与创新点
**Q3：大小模型协同的分工是什么？**
- 小模型：高效生成 Top‑K 根因候选，降低搜索空间与成本。
- 大模型：基于证据与候选给出解释与类型分类，提升可解释性。

**Q4：创新点有哪些？**
- 以候选 Top‑K + 证据汇聚构建可训练的结构化样本。
- 用统一输出 Schema 约束结果格式与可验证性。
- 轻量小模型基线 + LLM 解释模块的协同闭环。

**Q5：为什么要做 GRPO？**
- SFT 提供初始能力，但输出稳定性与一致性不足。
- GRPO 可通过奖励约束提升一致性、证据引用与格式合规性。

## 3. 数据与流程
**Q6：数据来源与结构是什么？**
- AIOps2020：故障清单 + 日志/指标/链路数据。
- 以故障窗口对齐多源观测，汇聚证据摘要。

**Q7：SFT 样本如何构建？**
- 输入：故障事件信息 + 证据汇总 + Top‑K 候选。
- 输出：根因组件 + 故障类型 + 解释依据。

**Q8：为什么使用时间窗口对齐？**
- 统一故障事件的证据时间范围，避免噪声干扰。
- 便于跨源融合与可复现的样本构建。

## 4. 实验与评估
**Q9：小模型评价指标是什么？**
- Top‑K 命中率（Hit@K），衡量候选覆盖率。

**Q10：大模型评价方式？**
- 输出格式合规性（Schema 校验）。
- 结果一致性与可解释性（规则化检查）。

**Q11：与传统方法相比的优势？**
- 更强的跨源证据聚合能力。
- 输出可解释、可校验，便于工程落地。

## 5. 系统工程
**Q12：系统模块如何划分？**
- 数据接入 → 时间对齐 → 证据聚合 → 候选生成 → LLM 推理 → 结果展示。

**Q13：系统如何保证稳定性？**
- 统一输入模板与输出 Schema。
- 端到端流程脚本化，易复现与排查。

## 6. 局限性与改进
**Q14：当前局限是什么？**
- 数据集标注有限，真实场景可能存在分布差异。
- 复杂跨服务故障仍依赖更高质量的链路数据。

**Q15：后续工作？**
- 扩展更长时间跨度的数据样本。
- 引入真实业务场景的在线反馈与持续优化。
